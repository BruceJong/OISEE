## 1. 概述

OISee 采用**单台阿里云 ECS + Docker Compose 编排 + 阿里云托管中间件（OSS/VOD/短信/AIGC）**的部署模式。这是为单人开发者优化的最低运维成本方案。

设计目标：
- **部署可重复**：一份 docker-compose.yml 描述完整生产环境
- **冷启动友好**：服务器宕机后可在 30 分钟内恢复（含数据）
- **零中转流量**：媒体走 OSS/CDN，不经 ECS
- **可观测**：日志可查、错误可见、备份可验

## 2. 阿里云资源清单

| 资源 | 规格建议 | 说明 |
| --- | --- | --- |
| ECS（计算） | 2核 4G + 50GB SSD（突发性能 t6 或共享标准 s6 起步） | 跑 Nginx + API + PG + Redis |
| EIP / 带宽 | 5 Mbps 按使用量计费 | 网站流量；视频走 CDN 不占 |
| 域名 | `oisee.com`（举例） | 阿里云域名 + 备案 |
| SSL 证书 | 阿里云免费 DV 证书 | 单域名 |
| OSS Bucket | 标准存储 + 公网读 | 存图片、视频、备份 |
| CDN | 阿里云 CDN | 加速 OSS，自定义域名 `cdn.oisee.com` |
| 短信服务 | 阿里云短信 | 注册验证码 |
| AIGC | DashScope（通义万相） | API 按量计费 |
| RAM 子账号 | 业务专用 AK/SK | 仅授权必要资源 |
| 云监控（基础） | 免费 | CPU / 内存 / 磁盘告警 |

> **特别提醒**：面向青少年的科普网站需**国内 ICP 备案**，建议早期就提交备案。备案期间可用临时管理后台域名（不对外）。

## 3. 网络与域名规划

### 3.1 域名

| 域名 | 用途 | 解析 |
| --- | --- | --- |
| `oisee.com` | 用户端首页 | A 记录 → ECS EIP |
| `www.oisee.com` | 用户端别名 | CNAME → oisee.com |
| `admin.oisee.com` | 管理端（可选） | A 记录 → ECS EIP |
| `api.oisee.com` | 后端 API（可选） | A 记录 → ECS EIP |
| `cdn.oisee.com` | OSS CDN 加速 | CNAME → 阿里云 CDN 域名 |

> **简化方案**：`oisee.com` 一个域名内分路径走（`/`、`/admin`、`/api`），不拆子域。这样 1 张证书就够，CORS 也无问题。下文以此简化方案为准。

### 3.2 端口与防火墙

ECS 安全组：
| 端口 | 协议 | 来源 | 用途 |
| --- | --- | --- | --- |
| 22 | TCP | 你的固定 IP / 跳板机 | SSH（限制源 IP） |
| 80 | TCP | 0.0.0.0/0 | HTTP（强制跳 HTTPS） |
| 443 | TCP | 0.0.0.0/0 | HTTPS |

容器内部端口（不对外暴露）：
- `:3000` API
- `:5432` PostgreSQL（仅 Docker 网络）
- `:6379` Redis

## 5. Docker Compose（生产）

`docker-compose.yml` 在仓库根目录：

```yaml
services:
  postgres:
    image: postgres:16-alpine
    container_name: oisee-postgres
    restart: unless-stopped
    environment:
      POSTGRES_USER: ${OISEE_DB_USER}
      POSTGRES_PASSWORD: ${OISEE_DB_PASSWORD}
      POSTGRES_DB: ${OISEE_DB_NAME}
    volumes:
      - /opt/oisee/data/postgres:/var/lib/postgresql/data
    healthcheck:
      test: ['CMD-SHELL', 'pg_isready -U ${OISEE_DB_USER}']
      interval: 10s
      timeout: 5s
      retries: 5
    networks:
      - oisee

  redis:
    image: redis:7-alpine
    container_name: oisee-redis
    restart: unless-stopped
    command: redis-server --appendonly yes --requirepass ${OISEE_REDIS_PASSWORD}
    volumes:
      - /opt/oisee/data/redis:/data
    healthcheck:
      test: ['CMD', 'redis-cli', '-a', '${OISEE_REDIS_PASSWORD}', 'ping']
      interval: 10s
    networks:
      - oisee

  api:
    build:
      context: .
      dockerfile: docker/api.Dockerfile
    container_name: oisee-api
    restart: unless-stopped
    env_file: /opt/oisee/.env.production
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    expose:
      - '3000'
    networks:
      - oisee

  nginx:
    build:
      context: .
      dockerfile: docker/nginx.Dockerfile
    container_name: oisee-nginx
    restart: unless-stopped
    ports:
      - '80:80'
      - '443:443'
    volumes:
      - /opt/oisee/data/nginx/ssl:/etc/nginx/ssl:ro
      - /opt/oisee/data/logs/nginx:/var/log/nginx
    depends_on:
      - api
    networks:
      - oisee

networks:
  oisee:
    driver: bridge
```

> **注意**：`web` 和 `admin` 的静态产物在 Nginx 镜像中（构建时多阶段拷贝），不需要单独的运行容器。

## 6. Dockerfile

### 6.1 `docker/api.Dockerfile`

```dockerfile
# ============ Stage 1: Build ============
FROM node:22-alpine AS builder
WORKDIR /app

RUN corepack enable

# 拷贝 monorepo 元数据与必要源码
COPY package.json pnpm-lock.yaml pnpm-workspace.yaml turbo.json* ./
COPY code/packages/shared/package.json ./code/packages/shared/
COPY code/packages/config/package.json ./code/packages/config/
COPY code/apps/api/package.json ./code/apps/api/

RUN pnpm install --frozen-lockfile

COPY code/packages/shared ./code/packages/shared
COPY code/packages/config ./code/packages/config
COPY code/apps/api ./code/apps/api

RUN pnpm --filter @oisee/shared build && \
    pnpm --filter @oisee/api prisma generate && \
    pnpm --filter @oisee/api build

# ============ Stage 2: Runtime ============
FROM node:22-alpine
WORKDIR /app

RUN corepack enable && apk add --no-cache tini

COPY --from=builder /app/code/apps/api/dist ./dist
COPY --from=builder /app/code/apps/api/node_modules ./node_modules
COPY --from=builder /app/code/apps/api/prisma ./prisma
COPY --from=builder /app/code/apps/api/package.json ./
COPY --from=builder /app/code/packages/shared/dist ./node_modules/@oisee/shared/dist
COPY --from=builder /app/code/packages/shared/package.json ./node_modules/@oisee/shared/

EXPOSE 3000
ENTRYPOINT ["/sbin/tini", "--"]
CMD ["sh", "-c", "npx prisma migrate deploy && node dist/main.js"]
```

### 6.2 `docker/nginx.Dockerfile`

```dockerfile
# ============ Stage 1: Build frontends ============
FROM node:22-alpine AS frontend-builder
WORKDIR /app

RUN corepack enable

COPY package.json pnpm-lock.yaml pnpm-workspace.yaml turbo.json* ./
COPY packages ./packages
COPY code/apps/web ./code/apps/web
COPY code/apps/cms ./code/apps/cms

RUN pnpm install --frozen-lockfile && \
    pnpm --filter @oisee/shared build && \
    pnpm --filter @oisee/web build && \
    pnpm --filter @oisee/cms build

# ============ Stage 2: Nginx ============
FROM nginx:1.27-alpine

COPY docker/nginx.conf /etc/nginx/nginx.conf
COPY --from=frontend-builder /app/code/apps/web/dist /usr/share/nginx/html/
COPY --from=frontend-builder /app/code/apps/cms/dist /usr/share/nginx/html/admin/

EXPOSE 80 443
```

## 7. Nginx 配置

`docker/nginx.conf`：

```nginx
user nginx;
worker_processes auto;
events { worker_connections 1024; }

http {
    include       mime.types;
    default_type  application/octet-stream;
    sendfile      on;
    keepalive_timeout 65;
    gzip          on;
    gzip_types    text/plain text/css application/javascript application/json image/svg+xml;
    gzip_min_length 1024;

    server_tokens off;

    # 限制请求体（防大文件攻击；真实上传走 OSS 直传）
    client_max_body_size 5m;

    # 通用安全头
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;

    # ===== HTTP → HTTPS =====
    server {
        listen 80;
        server_name oisee.com www.oisee.com;
        return 301 https://oisee.com$request_uri;
    }

    # ===== HTTPS 主入口 =====
    server {
        listen 443 ssl http2;
        server_name oisee.com;

        ssl_certificate     /etc/nginx/ssl/oisee.com.crt;
        ssl_certificate_key /etc/nginx/ssl/oisee.com.key;
        ssl_protocols       TLSv1.2 TLSv1.3;
        ssl_ciphers         HIGH:!aNULL:!MD5;
        ssl_session_cache   shared:SSL:10m;

        # 用户端 SPA
        location / {
            root /usr/share/nginx/html;
            try_files $uri /index.html;
            expires 0;
            add_header Cache-Control "no-store, must-revalidate";
        }

        # 用户端静态资源（带 hash 文件名）
        location /assets/ {
            root /usr/share/nginx/html;
            expires 1y;
            add_header Cache-Control "public, immutable";
        }

        # 管理端 SPA
        location /admin/ {
            alias /usr/share/nginx/html/admin/;
            try_files $uri /admin/index.html;
            expires 0;
            add_header Cache-Control "no-store, must-revalidate";
        }

        location /admin/assets/ {
            alias /usr/share/nginx/html/admin/assets/;
            expires 1y;
            add_header Cache-Control "public, immutable";
        }

        # API 反代
        location /api/ {
            proxy_pass         http://api:3000/api/;
            proxy_http_version 1.1;
            proxy_set_header   Host $host;
            proxy_set_header   X-Real-IP $remote_addr;
            proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header   X-Forwarded-Proto $scheme;
            proxy_read_timeout 60s;
        }

        # 健康检查
        location = /health {
            proxy_pass http://api:3000/api/v1/health;
            access_log off;
        }
    }

    # Swagger 文档（仅非生产暴露；生产建议彻底关闭）
    # 见 main.ts 的环境判断
}
```

## 9. 首次部署流程

### 9.1 ECS 初始化

```bash
# 1. 系统更新
apt update && apt upgrade -y

# 2. 安装 Docker（按官方文档）
curl -fsSL https://get.docker.com | bash

# 3. 安装基础工具
apt install -y git ufw fail2ban

# 4. 防火墙（与安全组冗余）
ufw allow 22/tcp
ufw allow 80/tcp
ufw allow 443/tcp
ufw enable

# 5. fail2ban（防 SSH 爆破）
systemctl enable --now fail2ban
```

### 9.2 部署应用

```bash
# 1. 创建目录结构
mkdir -p /opt/oisee/{data/{postgres,redis,nginx/ssl,logs},backups}

# 2. 克隆仓库
cd /opt/oisee
git clone <repo-url> repo
cd repo

# 3. 创建环境变量文件
nano /opt/oisee/.env.production    # 粘贴所有变量

# 4. 放入 SSL 证书
cp oisee.com.crt /opt/oisee/data/nginx/ssl/
cp oisee.com.key /opt/oisee/data/nginx/ssl/

# 5. 启动
docker compose --env-file /opt/oisee/.env.production up -d --build

# 6. 初次数据迁移（容器内自动执行 prisma migrate deploy，可手动确认）
docker compose logs api | grep "applied migration"

# 7. 灌入管理员账号（仅一次）
docker compose exec api node dist/scripts/seed-system.js
```

### 9.3 验证

- 浏览器访问 `https://oisee.com` → 用户端正常加载
- 浏览器访问 `https://oisee.com/admin` → 管理端登录页
- `curl https://oisee.com/api/v1/health` → `{ "code": 0, "data": { "status": "ok" } }`

## 10. 日常发布流程

```bash
# 在本地：推送代码到 main
git push origin main

# 在服务器：
cd /opt/oisee/repo
git pull
docker compose --env-file /opt/oisee/.env.production up -d --build

# 验证
docker compose ps                          # 查看容器健康
docker compose logs --tail=100 api         # 看启动日志
```

### 10.1 自动化（可选）

写个 `scripts/deploy.sh`，配合 GitHub Actions 在推送 main 时 SSH 到服务器执行：

```yaml
# .github/workflows/deploy.yml
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SSH_HOST }}
          username: ${{ secrets.SSH_USER }}
          key: ${{ secrets.SSH_KEY }}
          script: |
            cd /opt/oisee/repo
            git pull
            docker compose --env-file /opt/oisee/.env.production up -d --build
            docker image prune -f
```

> 单人项目可省略 CI/CD，纯手动也行；上 GitHub Actions 的好处是有部署日志可查。

### 10.2 零停机部署（可选）

MVP 用 `up -d` 重启即可（数秒不可用，对教育场景可接受）。如需零停机：
- 引入 blue-green：两套 API 容器轮换
- 增加复杂度，单人项目不推荐

## 11. 数据库迁移与回滚

### 11.1 应用迁移

每次部署，容器启动命令包含 `npx prisma migrate deploy`，自动应用未执行的迁移。

### 11.2 危险操作前的备份

部署涉及 Schema 变更的版本前，**手动触发一次备份**：

```bash
docker compose exec postgres pg_dump -U oisee oisee | gzip > /opt/oisee/backups/manual-$(date +%Y%m%d-%H%M%S).sql.gz
```

### 11.3 回滚

Prisma 不支持自动回滚迁移。如出现严重问题：
1. 回滚代码版本（`git checkout <prev-commit>`）
2. 恢复数据库（从最近备份）
3. 重启容器

> **教训**：避免破坏性 Schema 变更（如直接删列）。先增加新列、双写过渡、再删旧列。

## 12. 备份策略

### 12.1 数据库每日备份

`scripts/backup-db.sh`：

```bash
#!/usr/bin/env bash
set -euo pipefail

TS=$(date +%Y%m%d-%H%M%S)
BACKUP_FILE="/opt/oisee/backups/oisee-${TS}.sql.gz"

# 1. 导出
docker compose exec -T postgres pg_dump -U oisee oisee | gzip > "$BACKUP_FILE"

# 2. 上传 OSS（用 ossutil）
ossutil cp "$BACKUP_FILE" "oss://oisee-backup/db/oisee-${TS}.sql.gz"

# 3. 本地保留 7 天
find /opt/oisee/backups -name 'oisee-*.sql.gz' -mtime +7 -delete

# 4. OSS 上保留 30 天（用 OSS 生命周期规则）
```

### 12.2 Cron 定时

```bash
# crontab -e
0 2 * * * /opt/oisee/repo/scripts/backup-db.sh >> /opt/oisee/data/logs/backup.log 2>&1
```

### 12.3 OSS 生命周期

为 `oss://oisee-backup/db/` 路径设置规则：30 天后转低频存储或删除。

### 12.4 媒体备份

OSS 本身已有冗余（标准存储 11 个 9）。**不需要额外备份业务媒体**。但建议开启 OSS 版本控制，防止误删除。

### 12.5 恢复演练（季度）

每季度做一次"从备份恢复到测试服务器"的演练，验证备份有效性。

## 13. 监控与告警

### 13.1 基础监控（阿里云云监控）

| 指标 | 阈值 | 通知方式 |
| --- | --- | --- |
| CPU 持续 5min > 80% | ⚠️ | 短信 + 邮件 |

| 磁盘 > 85% | 🔴 | 短信（紧急） |
| 公网带宽 > 90% | ⚠️ | 短信 |
| 实例不可达 | 🔴 | 短信 |

### 13.2 应用日志

Docker 日志策略：
```yaml
# 在 docker-compose.yml 的每个服务下加：

  driver: json-file
  options:
    max-size: '10m'
    max-file: '5'
```

定期查看：
```bash
docker compose logs --tail=200 -f api
```

可在另一台机器或 [UptimeRobot](https://uptimerobot.com/) 配置：每 5 分钟检测 `https://oisee.com/health`，失败邮件告警。

## 14. 性能基线与扩展路径

### 14.1 单 ECS 承载能力（估算）

| 指标 | 2C4G 估算 |
| --- | --- |
| 并发用户数 | 500-1000（在线浏览） |
| API QPS | 100-200 |
| 同时进行的 AIGC 视频任务 | 2-3（worker concurrency） |

### 14.2 扩展路径（按瓶颈触发）

| 瓶颈 | 应对 |
| --- | --- |
| API CPU 打满 | 升级 ECS 规格（4C8G） |
| PG 慢查询 | 迁移到阿里云 RDS PostgreSQL |
| Redis 内存不够 | 迁移到阿里云 Redis |
| 视频流量打满 ECS | 已走 OSS+CDN，不应该出现 |
| AIGC 任务积压 | 拆独立 worker 容器，跨主机部署 |
| 多实例不一致（限流、缓存） | Redis 集中化 |

## 15. 安全运维清单

- [ ] SSH 仅密钥登录，禁用密码（`PasswordAuthentication no`）
- [ ] SSH 端口可改 22 以外（防扫描）
- [ ] 阿里云安全组限制 SSH 源 IP
- [ ] 启用 fail2ban
- [ ] 系统定期更新（`apt update && apt upgrade`，每月一次）
- [ ] Docker 镜像定期重建（拉最新 base）
- [ ] HTTPS 证书 90 天自动续签（阿里云免费证书可设置自动）
- [ ] 数据库密码 / API Key 半年轮换
- [ ] 备份恢复演练每季度一次
- [ ] 阿里云账号开启 MFA
- [ ] RAM 子账号仅最小权限（OSS 路径限定、DashScope 只读 + invoke）

## 16. 灾难恢复 RTO/RPO

| 场景 | RPO（数据丢失窗口） | RTO（恢复时间） | 处理 |
| --- | --- | --- | --- |
| ECS 实例重启 | 0 | 5 分钟 | 自动重启，容器自启 |
| ECS 实例损毁 | 24 小时（最近一次备份） | 1-2 小时 | 新建实例，拉取备份恢复 |
| 数据库损坏 | 24 小时 | 30 分钟 | 从备份恢复 |
| OSS 数据丢失 | 几乎不可能 | - | OSS 多副本冗余 |
| 阿里云区域故障 | 24 小时 | 数小时 | 切换到备用区域（需提前在另一区域备份） |

> MVP 单人项目接受 RPO=24h。如需更严格的 RPO（如 1 小时），需引入 PG WAL 流复制或托管 RDS。
