## 1. 概述

激励规则引擎（`RewardService`）是后端的核心业务模块，统一处理积分、勋章、解锁、等级四类联动逻辑。

设计目标：
- **集中处理**：所有积分/勋章变更只能通过 `RewardService`，散落在各处会导致数值不一致
- **事务一致性**：积分流水、勋章授予、解锁记录、用户累计积分必须原子写入
- **幂等**：同一行为多次上报不重复发奖
- **可观测**：每个行为返回结构化结果（积分增量、新勋章、新解锁、升级），前端可立即反馈
- **可演进**：规则参数（阈值、奖励值）集中可配，未来可挪到数据库

## 2. 核心抽象

### 2.1 行为事件（Action）

枚举所有触发激励的行为：

```ts
export enum RewardAction {
  EXPLORE_ITEM = 'EXPLORE_ITEM',
  LEARN_KP = 'LEARN_KP',

  COMPLETE_SCENE = 'COMPLETE_SCENE',
  DAILY_LOGIN = 'DAILY_LOGIN',

```

### 2.2 评估结果（RewardResult）

每次评估返回的结构化结果，可直接返回给前端用于反馈：

```ts
export interface RewardResult {
  pointsEarned: number;                       // 本次新增积分
  totalPoints: number;                        // 用户当前累计积分
  oldLevel: number;
  newLevel: number;
  levelUp: boolean;
  newBadges: BadgeAwarded[];                  // 本次新解锁的勋章
  newUnlocks: KnowledgePointUnlock[];         // 本次新解锁的知识点
}

export interface BadgeAwarded {
  id: string;
  slug: string;
  name: string;
  iconUrl: string;
  category: BadgeCategory;
}

export interface KnowledgePointUnlock {
  knowledgePointId: string;
  slug: string;
  name: string;
  difficulty: Difficulty;
}
```

### 2.3 服务接口

```ts
@Injectable()
export class RewardService {
  async onExploreItem(userId: string, itemId: string): Promise<RewardResult>;
  async onLearnKnowledge(userId: string, kpId: string): Promise<RewardResult>;
  async onCompleteExperiment(userId: string, experimentId: string): Promise<RewardResult>;
  async onCompleteScene(userId: string, sceneId: string): Promise<RewardResult>;
  async onDailyLogin(userId: string): Promise<RewardResult>;
}
```

每个方法内部统一调用 `evaluate(action, ctx)`，参数差异由 ctx 承载。

export const LEVEL_THRESHOLDS = [
  { level: 1, minPoints: 0,    title: '好奇宝宝' },

## 4. 评估流程

每次行为处理走以下统一流程：

```
1. 写行为持久化记录（如 KnowledgePointProgress）   ─┐
2. 计算积分（含幂等检查）                         │
3. 写 PointsLedger（带 idempotencyKey）            ├── 单个 Prisma 事务
4. 重新计算 User.totalPoints / level              │
5. 评估勋章规则                                   │
6. 评估解锁规则                                   │
7. 写 UserBadge / UserUnlock                      │
8. 更新场景探索度（如影响）                       │
                                                  ─┘
9. 组装 RewardResult 返回
```

### 4.1 幂等保证

每次行为生成幂等键：

| 行为 | idempotencyKey 格式 |
| --- | --- |
| `EXPLORE_ITEM` | `EXPLORE_ITEM:{userId}:{itemId}` |
| `LEARN_KP` | `LEARN_KP:{userId}:{kpId}` |
| `COMPLETE_EXPERIMENT` | `COMPLETE_EXPERIMENT:{userId}:{experimentId}` |
| `COMPLETE_SCENE` | `COMPLETE_SCENE:{userId}:{sceneId}` |
| `DAILY_LOGIN` | `DAILY_LOGIN:{userId}:{yyyy-MM-dd}` |

`PointsLedger.idempotencyKey` 设唯一约束 → 重复插入会触发 `P2002` → 视为已发奖，跳过积分发放，但仍执行后续勋章/解锁评估（确保即便此前漏检测的勋章也能补发）。

### 4.2 事务边界

```ts
async onLearnKnowledge(userId: string, kpId: string): Promise<RewardResult> {
  return this.prisma.$transaction(async (tx) => {
    // 1. 写学习记录
    await tx.knowledgePointProgress.upsert({
      where: { userId_knowledgePointId: { userId, knowledgePointId: kpId } },
      create: { userId, knowledgePointId: kpId },
      update: {},
    });

    // 2-9. 统一评估
    return this.evaluate(tx, {
      action: RewardAction.LEARN_KP,
      userId,
      kpId,
    });
  });
}
```

> **重要**：所有 reward 操作必须在事务内，避免出现"积分到账但勋章没发"的中间态。

## 5. 勋章评估

### 5.1 勋章定义结构

`Badge.conditions` 是结构化 JSON，规则引擎按结构解析：

```ts
// 探索类
{
  "type": "EXPLORATION",
  "scenario": "complete_all_l1_in_scene",
  "sceneId": "scene_kitchen"
}

// 学科类
{
  "type": "SUBJECT",
  "subject": "PHYSICS",
  "minKpCount": 10
}

// 实验类
{
  "type": "EXPERIMENT",
  "minExperimentCount": 5
}
```

### 5.2 评估器

每种 condition type 对应一个评估函数：

```ts
type BadgeEvaluator = (tx: PrismaTx, userId: string, conditions: any) => Promise<boolean>;

const evaluators: Record<string, BadgeEvaluator> = {
  EXPLORATION: async (tx, userId, c) => {
    if (c.scenario === 'complete_all_l1_in_scene') {
      const allL1 = await tx.knowledgePoint.count({
        where: {
          difficulty: 'L1',
          status: 'PUBLISHED',
          items: { some: { item: { sceneId: c.sceneId } } },
        },
      });
      const learned = await tx.knowledgePointProgress.count({
        where: {
          userId,
          knowledgePoint: {
            difficulty: 'L1',
            items: { some: { item: { sceneId: c.sceneId } } },
          },
        },
      });
      return allL1 > 0 && learned >= allL1;
    }
    return false;
  },

  SUBJECT: async (tx, userId, c) => {
    const count = await tx.knowledgePointProgress.count({
      where: {
        userId,
        knowledgePoint: { subject: c.subject },
      },
    });
    return count >= c.minKpCount;
  },

  EXPERIMENT: async (tx, userId, c) => {
    const count = await tx.experimentProgress.count({ where: { userId } });
    return count >= c.minExperimentCount;
  },
};
```

### 5.3 评估策略

为避免每次行为遍历所有勋章，采用**触发条件路由**：

| 行为 | 仅评估以下类型勋章 |
| --- | --- |
| `LEARN_KP` | `EXPLORATION`（场景完成度）、`SUBJECT`（学科达标） |
| `COMPLETE_EXPERIMENT` | `EXPERIMENT`、可能影响 `EXPLORATION`（如有"完成 N 个厨房实验"） |
| `EXPLORE_ITEM` | 暂无勋章直接触发，但保留扩展空间 |

实现：在 `Badge.conditions` 增加 `triggers: ['LEARN_KP']` 字段，评估时只遍历目标 badges：

```ts
const candidates = await tx.badge.findMany({
  where: {
    NOT: { users: { some: { userId } } },     // 用户尚未拥有
    conditions: { path: ['triggers'], array_contains: [action] },
  },
});
```

> PG 的 JSON 查询，性能在勋章数量 < 100 完全够用。

### 5.4 写入勋章

满足条件的勋章插入 `UserBadge`：

```ts
const created = await tx.userBadge.createMany({
  data: awardedIds.map((badgeId) => ({ userId, badgeId })),
  skipDuplicates: true,
});
```

## 6. 解锁评估

### 6.1 触发时机

知识点解锁的触发条件：
1. 用户学习了新知识点（可能完成了某物品的全部 L1 → 解锁 L2）
2. 用户获得了新勋章（可能补足 L2 解锁所需的"≥3 枚勋章"）
3. 用户完成了实验（可能补足 L3 解锁所需的"≥1 枚实验勋章"）

简化处理：**只在勋章评估后触发一次解锁评估**，因为勋章评估覆盖了学习/实验等所有副作用。

### 6.2 解锁算法

对每个物品的每个用户当前难度+1层级，检查是否满足条件：

```ts
async evaluateUnlocks(tx, userId): Promise<KnowledgePointUnlock[]> {
  const userBadges = await tx.userBadge.findMany({ where: { userId } });
  const badgeCount = userBadges.length;
  const expBadgeCount = await tx.userBadge.count({
    where: { userId, badge: { category: 'EXPERIMENT' } },
  });

  // 找到所有"用户已学完该物品全部 L1，但 L2 尚未解锁"的物品
  const itemsForL2 = await this.findItemsReadyForUnlock(tx, userId, 'L1', 'L2');
  const itemsForL3 = await this.findItemsReadyForUnlock(tx, userId, 'L2', 'L3');

  const unlocks: KnowledgePointUnlock[] = [];

  if (badgeCount >= UNLOCK_RULES.L2.requireBadges.count) {
    for (const item of itemsForL2) {
      const kps = await tx.knowledgePoint.findMany({
        where: {
          difficulty: 'L2',
          items: { some: { itemId: item.id } },
          NOT: { /* 已解锁 */ unlocks: { some: { userId } } },
        },
      });
      for (const kp of kps) {
        await tx.userUnlock.create({
          data: { userId, knowledgePointId: kp.id, reason: 'rule' },
        });
        unlocks.push({ ...kp });
      }
    }
  }

  if (expBadgeCount >= UNLOCK_RULES.L3.requireBadges.count) {
    // 同理 L3
  }

  return unlocks;
}
```

> 这是**示意逻辑**，实际实现需要根据精确的 PRD 规则展开。`findItemsReadyForUnlock` 是辅助方法。

### 6.3 锁定态查询

前端调物品详情时，后端为每个知识点判断 `locked: boolean`：

```ts
function isLocked(kp, user, userUnlocks): boolean {
  if (!user) return false;        // 匿名用户：默认场景内全部可见但不计数
  if (difficultyLevel(kp.difficulty) <= difficultyLevel(user.difficulty)) return false;
  return !userUnlocks.has(kp.id);
}
```

> 用户难度 `user.difficulty` 决定基础可学层；超过的需要 `UserUnlock` 记录解锁。

## 7. 场景探索度更新

PRD §3.3：场景探索完成度 = 该场景内知识点的总学习程度。

### 7.1 派生公式

```
completionRate = COUNT(已学知识点 in 场景) / COUNT(全部已发布知识点 in 场景)
```

### 7.2 更新时机

`LEARN_KP` 触发后，找到该知识点所属的场景（通过 `Item → Scene`），重新计算 `SceneProgress.completionRate`。

注意：一个知识点可能挂在多个物品（属于不同场景），需要对每个相关场景独立更新。

```ts
const scenes = await tx.scene.findMany({
  where: { items: { some: { knowledgePoints: { some: { knowledgePointId: kpId } } } } },
  select: { id: true },
});

for (const scene of scenes) {
  const { total, learned } = await this.computeSceneCompletion(tx, userId, scene.id);
  await tx.sceneProgress.upsert({
    where: { userId_sceneId: { userId, sceneId: scene.id } },
    create: { userId, sceneId: scene.id, completionRate: learned / total },
    update: {
      completionRate: learned / total,
      completedAt: learned >= total ? new Date() : null,
    },
  });

  // 检查是否首次完成场景 → 触发 COMPLETE_SCENE 奖励（递归调用，但用幂等键保护）
  if (learned >= total && !alreadyCompleted) {
    await this.awardSceneCompletionPoints(tx, userId, scene.id);
  }
}
```

## 8. 等级评估

```ts
function updateLevel(tx, userId, oldPoints, newPoints) {
  const oldLevel = levelFromPoints(oldPoints);
  const newLevel = levelFromPoints(newPoints);
  if (newLevel !== oldLevel) {
    await tx.user.update({
      where: { id: userId },
      data: { totalPoints: newPoints, level: newLevel },
    });
  } else {
    await tx.user.update({
      where: { id: userId },
      data: { totalPoints: newPoints },
    });
  }
  return { oldLevel, newLevel };
}
```

升级时是否单独发奖（如送称号）：MVP 不单独发，仅前端展示升级动画。

## 9. 完整评估示例（学习知识点）

```ts
async evaluate(tx: PrismaTx, ctx: { action: 'LEARN_KP'; userId: string; kpId: string }): Promise<RewardResult> {
  const { userId, kpId } = ctx;
  const user = await tx.user.findUniqueOrThrow({ where: { id: userId } });
  const oldPoints = user.totalPoints;

  // 1. 积分（幂等）
  const key = `LEARN_KP:${userId}:${kpId}`;
  let pointsEarned = 0;
  try {
    await tx.pointsLedger.create({
      data: {
        userId,
        delta: POINTS.LEARN_KP,
        reason: 'LEARN_KP',
        refType: 'kp',
        refId: kpId,
        idempotencyKey: key,
      },
    });
    pointsEarned = POINTS.LEARN_KP;
  } catch (e) {
    if (e.code !== 'P2002') throw e;        // 非幂等冲突的错误抛出
  }

  // 2. 更新积分总数与等级
  const newPoints = oldPoints + pointsEarned;
  const { oldLevel, newLevel } = await updateLevel(tx, userId, oldPoints, newPoints);

  // 3. 更新场景探索度
  await this.updateSceneProgress(tx, userId, kpId);

  // 4. 评估勋章
  const newBadges = await this.evaluateBadges(tx, userId, 'LEARN_KP');

  // 5. 评估解锁（勋章评估后再评估，确保新勋章影响解锁判断）
  const newUnlocks = await this.evaluateUnlocks(tx, userId);

  return {
    pointsEarned,
    totalPoints: newPoints,
    oldLevel,
    newLevel,
    levelUp: newLevel > oldLevel,
    newBadges,
    newUnlocks,
  };
}
```

## 10. 数据一致性保护

### 10.1 累计积分的最终一致性

理论上 `User.totalPoints` 可由 `SUM(PointsLedger.delta)` 派生。为应对极端不一致：
- 提供管理端工具：`POST /api/v1/admin/users/:id/recompute-points`
- 重算时全表扫一次该用户 `PointsLedger`，覆盖 `totalPoints` 和 `level`

### 10.2 勋章重算

类似地，提供 `POST /api/v1/admin/users/:id/recompute-rewards`，重新跑一遍全部勋章 + 解锁评估。

### 10.3 内容变更影响

运营修改内容（如把某 L2 知识点改为 L1）后，已学用户的解锁可能受影响。MVP 不做"内容变更时的级联重算"，由运营在发布前规避此类破坏性修改。CMS 给出**警告提示**：「修改难度可能影响用户解锁状态，是否继续？」

## 11. 性能考量

| 场景 | 评估耗时 | 优化 |
| --- | --- | --- |
| 学习一个知识点 | < 100ms（含 5-10 次小查询） | 索引已覆盖，无需优化 |
| 评估全部勋章 | 候选集 < 10 时 < 50ms | 用 triggers 字段过滤候选 |
| 评估解锁 | 用户已学物品数 × 2（L2 + L3） | 大量学习时可能 200ms+，可异步化 |
| 我的书包数据聚合 | 单次 < 200ms | `Promise.all` 并行查询 |

> 当前规模下完全可同步评估。未来内容量大时，可把"解锁评估"挪到 BullMQ 异步执行，前端通过 SSE 推送。

## 12. 测试覆盖

规则引擎是**强约束业务逻辑**，必须单元测试。

### 12.1 测试用例清单

| 用例 | 期望 |
| --- | --- |
| 首次学习知识点 | +10 分，写流水 |
| 重复学习同一知识点 | 0 分增加，但不报错 |
| 学完物品全部 L1 知识点 | 触发"厨房小达人"勋章（若该物品是厨房唯一物品） |
| 学完 10 个物理知识点 | 触发"物理启蒙者"勋章 |
| 拥有 3 枚勋章 + 学完物品全部 L1 | L2 知识点解锁 |
| 拥有 1 枚实验勋章 + 学完物品全部 L2 | L3 知识点解锁 |
| 积分跨越等级阈值 | `levelUp: true`，level 更新 |
| 同时触发多个勋章 | 全部返回，事务一致 |
| 完成场景所有知识点 | 触发 `COMPLETE_SCENE` 奖励（+100） |
| 重复触发"完成场景" | 不重复发奖 |

### 12.2 测试基建

- 用 SQLite 内存版（Prisma 测试模式）或独立的测试 PG schema
- 每个测试用例独立的事务回滚
- 工厂函数构造用户、场景、物品、知识点、勋章
- 断言：`PointsLedger` 行数、`UserBadge` 集合、`User.totalPoints`、`RewardResult` 结构
