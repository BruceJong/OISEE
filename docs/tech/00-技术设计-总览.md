## 1. 文档目的与范围

本套技术设计文档（以下简称 "TD"）回答的是「**怎么做**」的问题。PRD 已明确「做什么」，TD 在此基础上落地：

- 技术选型与版本锁定
- 系统架构与部署形态
- 数据模型与 API 契约
- 关键技术问题的处理方案（2.5D 渲染、AIGC 工作流、激励规则引擎等）
- 工程化与开发规范

TD **不**包含具体业务逻辑（如积分数值、勋章具体清单等），这些以 PRD 与运营配置为准。

## 3. 项目约束（来自 PRD 与开发条件）

| 约束 | 描述 | 对技术决策的影响 |
| --- | --- | --- |
| 单人全流程开发 | 设计、前后端、部署一人完成 | 倾向一致性强的技术栈、AI 友好的约定式框架 |
| 阿里云个人 ECS 部署 | 单机部署，资源有限 | Docker Compose 单机编排，避免 K8s 等过度方案 |
| 目标终端：PC + 平板 | 响应式，平板需流畅 | 前端使用静态资源 + CDN，2.5D 用 CSS 而非 WebGL |
| MVP 仅 1 个场景 | 内容有限但骨架完整 | 数据模型一次到位，避免后期大改 |
| 全量用户为未成年人 | 合规硬约束 | 监护人同意、内容审核、第三方登录合规需在设计上预留 |
| AIGC 内容生产 | 文生图 + 文生视频，耗时异步 | 引入任务队列（BullMQ + Redis） |

## 9. 命名与版本约定

- **代码仓库**：单一仓库 `oisee/`，git 分支策略采用 trunk-based，主分支 `main` 自动部署到 prod。
- **NPM 包前缀**：内部包统一 `@oisee/*`（如 `@oisee/shared`）。
- **API 版本**：URL 内置版本号 `/api/v1/*`，未来不兼容升级走 `/api/v2/*`。
- **环境变量**：统一前缀 `OISEE_*`，由根目录 `.env` 派生到各 app；生产环境通过 Docker secrets / `.env.production` 注入。
- **数据库迁移**：`prisma migrate dev` 本地生成，`prisma migrate deploy` 生产应用，迁移文件入库。

## 10. 风险与对策

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| 单机部署单点故障 | 站点不可用 | 接受为 MVP 风险，每日备份至 OSS；后期可升级为 RDS + ECS 集群 |
| 视频带宽打满 ECS | 网站卡顿 | 视频走阿里云 VOD + CDN，不经 ECS |
| AIGC 调用失败率/耗时不可控 | 内容生产体验差 | 队列重试 + 任务状态可视化 + 兜底手动上传通道 |
| 单人开发节奏不稳 | 进度延期 | 严格按 MVP 范围切片，禁止二期功能渗透 |
| 未成年人合规细则未定 | 上线前阻塞 | 监护人同意弹窗 + 第三方平台对接预留接口，上线前补法务确认 |

## 11. 后续迭代建议（非 MVP）

- **二期功能**：知识点小测验、实验照片打卡、材料包商城
- **架构升级路径**：拆 RDS、引入对象存储签名直传、独立部署 CMS、引入 CI/CD

- **多场景内容上量后**：评估是否拆分搜索为 Meilisearch / Typesense
